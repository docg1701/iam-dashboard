# Story 1.4: Update API Layer for Direct Agent Integration

## Status
Approved

## Story
**As a** frontend developer,
**I want** API endpoints that call agents directly,
**so that** the UI can interact with the new agent architecture without changing request/response contracts.

## Acceptance Criteria

1. **API Endpoint Updates**: All document processing and questionnaire endpoints updated to call agents directly instead of queuing Celery tasks
2. **Response Schema Preservation**: API responses maintain identical JSON schemas to ensure frontend compatibility
3. **Error Response Consistency**: Error handling and HTTP status codes remain consistent with previous API behavior
4. **Synchronous Processing**: Endpoints return results directly from agent processing without async task management
5. **API Documentation**: Updated OpenAPI documentation reflects new agent-based processing while maintaining contract compatibility

## Dev Notes

### Previous Story Insights
The PDFProcessorAgent and QuestionnaireAgent implementations from stories 1.2 and 1.3 have established successful agent integration patterns. Both agents are fully operational and integrated with the AgentManager system via dependency injection. The agents successfully process documents and generate questionnaires while maintaining compatibility with existing database schemas and providing comprehensive error handling.

### Architecture Context
This story implements the API layer migration as part of the Direct Replacement Strategy, updating FastAPI endpoints to call agents directly while maintaining zero breaking changes to frontend contracts and identical request/response schemas.

### Key Technical Details

**Data Models**: No database schema changes required - agents use existing document and questionnaire_drafts table structures. API endpoints maintain identical request/response models to preserve frontend compatibility [Source: architecture/data-models-and-schema-changes.md#database-schema-strategy-zero-modification-approach]

**API Specifications**: 
- **Endpoint Migration**: Replace Celery task queuing with direct agent calls using AgentManager integration [Source: architecture/api-design-and-integration.md#api-endpoint-migration-strategy]
- **Response Schema Preservation**: Maintain identical DocumentProcessResponse and QuestionnaireGenerationResponse schemas [Source: architecture/api-design-and-integration.md#response-schema-preservation]
- **Error Handling**: Implement consistent error response format with agent-specific error mapping [Source: architecture/api-design-and-integration.md#error-handling-integration]
- **Performance Requirements**: Document processing ≤30s, questionnaire generation ≤15s, agent management ≤2s [Source: architecture/api-design-and-integration.md#performance-considerations]

**Component Specifications**:
- **AgentManager Integration**: Use existing dependency injection patterns with Container.agent_manager provider [Source: architecture/source-tree-integration.md#file-integration-details]
- **API Endpoints**: Update existing `/documents/process` and questionnaire endpoints to use agent_manager.get_agent() calls [Source: architecture/api-design-and-integration.md#api-endpoint-migration-strategy]
- **Error Middleware**: Implement AgentError exception handling with proper HTTP status code mapping [Source: architecture/api-design-and-integration.md#error-handling-integration]

**File Locations**: 
- Document API Updates: `app/api/documents.py` [Source: architecture/source-tree-integration.md#file-integration-details]
- Questionnaire API Updates: `app/api/questionnaire.py` [Source: architecture/source-tree-integration.md#file-integration-details]
- New Admin API: `app/api/admin.py` for agent management endpoints [Source: architecture/source-tree-integration.md#target-project-structure-with-agents]
- Container Updates: `app/containers.py` to ensure agent_manager provider is available [Source: architecture/source-tree-integration.md#file-integration-details]

**Technical Constraints**:
- FastAPI>=0.116.0 with Agno>=1.7.5 framework integration [Source: architecture/tech-stack-alignment.md#technology-integration-matrix]
- Maintain existing dependency-injector>=4.48.1 patterns for AgentManager injection [Source: architecture/tech-stack-alignment.md#technology-integration-matrix]
- Preserve PostgreSQL+pgvector integration without schema changes [Source: architecture/data-models-and-schema-changes.md#database-schema-strategy-zero-modification-approach]
- Performance parity: agent processing within 110% of current response times [Source: architecture/api-design-and-integration.md#performance-considerations]

**Testing Requirements**: 
- Unit tests location: `tests/unit/test_api/` for API endpoint tests [Source: architecture/testing-strategy.md#unit-testing-strategy]

- Coverage requirement: Minimum 80% code coverage for all API endpoint changes [Source: architecture/testing-strategy.md#testing-approach-overview]

**Integration Verification Requirements**:
- **Frontend Compatibility**: Existing NiceGUI components continue working without modification after API updates [Source: architecture/api-design-and-integration.md#api-endpoint-migration-strategy]

## Testing

**Test Standards**:
